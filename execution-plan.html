1How to run this plan

Working model

  • One milestone ≈ one focused Claude Code session in this repo (M2 and M7 may take two). Each milestone block below ends with a one-line kickoff prompt to paste into a fresh session.
  • Every milestone ends at a gate GATE: concrete acceptance criteria, checked before moving on. Gates from M2 onward include a fixture re-run, so regressions surface immediately.
  • Vertical slice first. M0→M1→M2 is the critical path and already delivers the LinkedIn post's core output (cited, severity-rated, scored findings). Everything after is enrichment, and M3/M4 can run in parallel.
  • Commit per task (T2.3: add filter.md), tag per milestone (m2-mvp). The repo is not yet a git repo — M0 fixes that.

Dependency graph

M0 scaffolding └─ M1 registry ──── M2 single-screen MVP ─┬─ M3 measure.py ──┐ ├─ M4 annotate.py ─┼─ M6 HTML report ── M7 calibration └─ M5 flow mode ───┘ M3 and M4 are independent of each other (parallelizable). M5 needs only M2. M6 needs M4 (annotated images) and benefits from M5. M7 hardens everything and is last.

M1 — Framework reference registry

1–1.5 sessionsdepends: M0
T1.1
Registry format spec
Write references/frameworks/_format.md: YAML frontmatter (prefix, framework, source, tier, category), then three required sections — Principles (table: ID | name | one-line definition), Screenshot-observable symptoms (per principle: 2–4 concrete visual cues — this is what makes the model find things), Worked example (1–2 findings in the output schema). See Appendix A.2 for the skeleton.
T1.2
Author Tier 1 files (6)
gestalt.md (GST-01…07), wcag.md (native SC numbers, AA focus, the 9 screenshot-auditable criteria), cognitive-laws.md (HICK/FITTS/MILL/JAKOB/PEAK — include the lawsofux misapplication warnings, e.g. Miller's ≠ "max 7 menu items"), gerhardt-powals.md (GP-01…10), content-design.md (CNT-01…10), cialdini.md (CLD-01…06 + DARK-01…06 dark-pattern checks: fake scarcity, manufactured social proof, pre-checked commitments, confirmshaming, forced continuity, hidden costs).
T1.3
Author Tier 1–2 files (4)
nielsen.md (NLS-01…10, mark which need flow context), norman.md (NRM-01…07 + NRM-GE/GV gulfs), shneiderman.md (SHN-01…08), tognazzini.md (TOG-01…19 — symptoms only for the ~12 screenshot/flow-relevant ones; mark the timed-study ones Tier 3).
T1.4
Author Tier 3 + custom files (2)
fogg.md (FOGG-M/A/P with the diagnostic logic: behavior missing → which element is absent; every finding is a hypothesis with a validation method). recurring-failures.md (RF-01…12) seeded from Qualia's controlled list: false affordance, dead-end state, unguarded destructive action, vague CTA label ("Submit" vs outcome-named), silent success, missing empty state, breadcrumb gap, choice paralysis, trust-signal gap, feedback latency, confirmation trap, recall trap.
T1.5
Machine-readable index
scripts/build_registry.py (stdlib): parses frontmatter + principle tables from all framework files → references/registry.json (ID → {name, framework, tier, category}). Re-run whenever files change; this is what citation validation checks against.
Gate M1
  • 12 framework files + _format.md; every file parses through build_registry.py with zero warnings; all IDs unique.
  • Spot-check 10 random principle definitions against the source URLs (the framework-review notes from this session are the cross-check).
  • Every file ≤120 lines; every principle in Tier 1 files has ≥2 observable symptoms.

M2 — Single-screen MVP (the vertical slice)

1.5–2 sessionsdepends: M1
T2.1
scoring.md
Severity table (verbatim Qualia definitions, weights 4/3/2/1), reach bands (1/1.5/2/3), importance formula, six category weights (30/15/15/15/15/10), overall bands (0–49/50–65/66–79/80–89/90+), anti-compression rule, and per-category 90/50/20 anchors — written fresh per category since Qualia's anchors are code-audit-specific (e.g. Accessibility 90 = "no measured contrast failures, all inputs visibly labeled, targets ≥ platform minimum").
T2.2
filter.md
The four gates verbatim (Blocker / Standard / Clutter / Steelman) + 4 worked examples of dropped candidates, one per gate (e.g. Standard gate drops "breadcrumbs would help" on a standard form page). The skill applies this to every candidate before it becomes a finding.
T2.3
report-spec.md
Full findings.json schema (Appendix A.1) — including clean_checks (what passed, enabling no-finding honesty) — the ≤60-line summary.md template, and the validation checklist: word counts, box sanity (ymin<ymax, localized↔box required, screen-level↔null), citations resolve in registry.json, hypotheses excluded from scores, ≤5 findings per category.
T2.4
SKILL.md
Frontmatter (name ux-audit, description with trigger phrases: "audit this design / review this screenshot / UX audit"). Body: (1) intake — one AskUserQuestion round, 4 questions with exact wording and defaults (goal free-text; persona default "first-time visitor"; platform mobile-touch/desktop/responsive; stage live/hi-fi/wireframe); (2) mode detection; (3) progressive-disclosure map — which reference files to load per mode/tier; (4) per-category analysis passes with the principle-ID-required rule; (5) filter pass; (6) scoring + One Big Thing (four rules); (7) validation checklist; (8) emit findings.json + summary.md. ≤150 lines.
T2.5
End-to-end dry runs
Run the skill on 3 real screenshots: one obviously flawed (a busy checkout or dashboard), one polished production app (tests no-finding honesty + Standard gate), one mobile screen. Iterate SKILL.md/reference wording until output passes the validation checklist without hand-holding. Capture these runs — they become fixture seeds in M7.
Gate M2
  • A fresh session with only /ux-audit + screenshot produces schema-valid findings.json + summary.md with zero manual fixes.
  • 100% of findings cite IDs that resolve in registry.json; all three text-field word limits hold.
  • The polished-app run produces ≤3 findings and scores ≥80 (no manufactured issues); the flawed run produces ≥8 findings spanning ≥4 categories with a falsifiable One Big Thing.
  • Scores across the three runs span ≥25 points (anti-compression smoke test).
Kickoff: Execute M2 from execution-plan.html — write scoring.md, filter.md, report-spec.md, SKILL.md, then dry-run on 3 screenshots I'll provide.

M3 — Deterministic measurement

1 sessiondepends: M2 · parallel with M4
T3.1
measure.py — auto pass
CLI per Appendix A.3. v1 strategy (honest over clever): downscale → color-quantize → detect high-contrast-edge text regions → for each region report fg/bg color pair + WCAG ratio + px height. Report "unmeasurable" for ambiguous regions (gradients, imagery) rather than guessing. Also: image dims, smallest detected text height, smallest isolated clickable-looking region.
T3.2
measure.py — region probe
--region ymin,xmin,ymax,xmax --probe contrast|size (0–1000 coords): the model requests measurements for specific elements during analysis instead of estimating. This is the primary interface; the auto pass is best-effort seeding.
T3.3
Synthetic tests
scripts/tests/test_measure.py: generate PNGs with known fg/bg pairs (e.g. #767676 on #fff = 4.54:1) and known box sizes; assert ratio within ±0.1 and size within ±2px. Runner: .venv/bin/python -m pytest or plain asserts via python scripts/tests/test_measure.py.
T3.4
Skill integration
SKILL.md step 2.5: run auto pass before analysis, inject output as measurements; add the hard rule — numeric WCAG claims require a measured value; otherwise phrase as "appears low-contrast — verify" at medium severity max. Findings carry evidence: "measured: …".
Gate M3
  • Synthetic tests pass; auto pass on the 3 M2 screenshots returns ≥5 plausible region measurements each with zero crashes.
  • Re-run one M2 dry-run: every numeric contrast claim now carries measured evidence.
Kickoff: Execute M3 from execution-plan.html — implement measure.py to the CLI contract in scripts/README.md, with synthetic tests.

M4 — Annotation

1 sessiondepends: M2 · parallel with M3
T4.1
annotate.py
CLI per Appendix A.3. Convert 0–1000 boxes to px; rounded rectangles color-coded by severity (crit #ff6b6b / high #ffa94d / med #ffd43b / low #74c0fc); numbered circular badge at top-left corner matching finding ID; 2px outline + subtle dark halo for legibility on any background. Draw lowest severity first so critical sits on top; stagger badges when boxes overlap (offset along the top edge).
T4.2
Box validation in-script
Refuse invalid boxes (ymin≥ymax, out of range, area >80% of screen → reclassify as screen-level) — log dropped boxes to stderr and continue. The skill's validation checklist references the same rules so bad boxes get fixed upstream, not silently dropped.
T4.3
Visual check
Annotate the M2 dry-run outputs; eyeball that badges are legible at 100% zoom and boxes sit on the elements the findings describe. Fix offsets/colors as needed.
Gate M4
  • annotated/screen-N.png produced for each M2 run; badge numbers match finding IDs; invalid-box handling verified with a hand-corrupted findings.json.
Kickoff: Execute M4 from execution-plan.html — implement annotate.py to the CLI contract and verify on the M2 outputs.

M5 — Flow mode (Tier 2)

1 sessiondepends: M2
T5.1
Flow prompt section in SKILL.md
2+ images → confirm ordering with the user, then: per-screen Tier 1 pass (as M2) + transition pass loading the Tier 2 references — cross-screen consistency (NLS-04, SHN-01: same action, same UI?), status feedback across transitions (NLS-01: does screen N+1 acknowledge what happened on N?), error/undo paths (NLS-03/05/09, SHN-06), memory load between screens (SHN-08, GP), peak-end check on the final screen (PEAK-01: does the flow end on confirmation/accomplishment or a dead end?).
T5.2
Schema + report additions
In report-spec.md: screen_index required in flow mode (null = cross-screen finding, which sets reach ≥ screen count); new top-level flow block — transitions[] (from→to, verdict, note), missing_states[] (expected loading/error/confirmation screens absent from the sequence), peak_end note. Update summary.md template with a flow section.
T5.3
Dry run on a real flow
3–5 ordered screenshots of a real signup or checkout flow. Verify cross-screen findings appear with correct reach values and the flow block is populated.
Gate M5
  • Flow dry-run yields ≥1 valid cross-screen finding and a populated flow block; single-screen mode still passes the M2 gate (no regression).
Kickoff: Execute M5 from execution-plan.html — add flow mode to SKILL.md and report-spec.md, then dry-run on a flow I'll provide.

M6 — HTML report

1 sessiondepends: M4 (+M5 for flow sections)
T6.1
report-template.html
Single-file shell (inline CSS/JS, zero network requests): header scorecard — six category bars + overall dial + score-band label; One Big Thing callout; annotated screenshots with finding cards alongside (click a card → highlight the box via overlay coordinates from findings.json, no image regeneration); findings table sortable by importance/severity/category; separate clearly-labeled Hypotheses section with validation methods; methodology footer (frameworks applied, tiers run, measurement coverage, model + date).
T6.2
build_report.py
CLI per Appendix A.3: inject findings.json + base64-encoded annotated PNGs into the template via {{PLACEHOLDER}} substitution (stdlib only). Output report.html, typically 2–8 MB — acceptable for a single-file artifact.
T6.3
Serve step in SKILL.md
Final skill step: write outputs to audits/<slug>-<date>/, start python3 -m http.server PORT --bind 0.0.0.0 if not already running, curl for 200, hand over http://clawbot.crane-cod.ts.net:PORT/.
Gate M6
  • Report for an M2 run and the M5 flow run renders correctly over Tailscale; card-click highlighting works; no external requests (verify with devtools/network or grep for http src).
Kickoff: Execute M6 from execution-plan.html — build report-template.html and build_report.py, render the existing dry-run outputs.

M7 — Calibration & hardening

1.5 sessionsdepends: M3–M6
T7.1
Golden fixtures (5)
fixtures/<name>/ each with screenshots + expected.json (curated findings with tolerance notes: must-find / should-find / must-NOT-find): (a) flawed checkout screen, (b) polished production screen — the no-finding-honesty test, its must-NOT-find list is the important part, (c) mobile screen with sub-24px targets — exercises measure.py, (d) 4-screen flow with a seeded inconsistency (button renamed mid-flow) and a missing confirmation screen, (e) dark-pattern pricing page (fake countdown, pre-checked add-on, confirmshame copy). Seed (a)–(c) from the M2 dry-runs.
T7.2
Fixture check procedure
scripts/check_fixtures.py diffs an audit's findings.json against expected.json → matched / missed must-finds / forbidden hits / score deltas vs recorded baseline (±10 tolerance). Document in CLAUDE.md: run after any prompt or reference change. Test the test: deliberately break filter.md (delete the Standard gate), re-run, confirm the polished fixture now fails on forbidden hits; revert.
T7.3
Re-audit mode
Skill accepts a prior findings.json + new screenshots → delta report: fixed (finding no longer reproduces), regressed, new, unchanged. Match findings by principle ID + screen + box overlap (IoU > 0.3) with fuzzy text fallback. New delta block in report-spec.md; delta section in the HTML report.
T7.4
Tier 3 opt-in pass
Intake gains an optional 5th question ("Also analyze conversion/persuasion? Findings will be hypotheses to test, not scored issues."). Loads fogg.md + cialdini.md Tier 3 sections; outputs to hypotheses only, each with a concrete validation method ("instrument funnel step 3", "A/B the CTA copy").
T7.5
Docs & wrap
Finish README (usage, examples, fixture workflow); update CLAUDE.md with the maintenance loop — recurring real-world failures get promoted into recurring-failures.md with new RF IDs (the post author's "custom checks" practice).
Gate M7 — Definition of Done
  • All 5 fixtures pass: every must-find found, zero forbidden hits, scores within ±10 of baseline.
  • The broken-filter canary test demonstrably fails the suite (then passes after revert).
  • Re-audit of fixture (a) with one issue visually fixed reports exactly that finding as fixed.
  • Full deliverable parity with the LinkedIn post: severity-rated findings ✓ heuristic citations ✓ recommendations ✓ annotated screenshots ✓ structured report ✓ — plus scores, measured evidence, and tier-honest hypotheses it didn't have.
Kickoff: Execute M7 from execution-plan.html — build the fixture suite and check script first, then re-audit mode, then the Tier 3 opt-in.

A.3 Script CLI contracts

# measure.py — deterministic visual measurements (Pillow)
.venv/bin/python scripts/measure.py IMG [IMG...] [--json out.json]
.venv/bin/python scripts/measure.py IMG --region 120,640,210,980 --probe contrast|size
# → {"screens":[{"file","width","height","regions":[
#      {"box_2d":[...],"kind":"text","fg":"#8a8f98","bg":"#6b7280",
#       "contrast":2.8,"height_px":14}|{"box_2d":[...],"kind":"unmeasurable"}]}]}

# annotate.py — draw findings onto screenshots
.venv/bin/python scripts/annotate.py findings.json --images s0.png s1.png --out annotated/
# → annotated/screen-0.png ...  (invalid boxes logged to stderr, never guessed)

# build_report.py — single-file HTML report
.venv/bin/python scripts/build_report.py findings.json --annotated annotated/ \
    --template assets/report-template.html --out report.html

# build_registry.py / check_fixtures.py
.venv/bin/python scripts/build_registry.py            # frameworks/*.md → registry.json
.venv/bin/python scripts/check_fixtures.py fixtures/  # diff audits vs expected.json
Why freeze these now M3/M4/M6 can be built in parallel sessions only if the interfaces don't move. Any contract change requires updating this appendix + scripts/README.md in the same commit.
