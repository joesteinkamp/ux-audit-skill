1Goal & scope

Build a Claude Code skill (/ux-audit) that reproduces — and improves on — the workflow from the LinkedIn post: upload screenshots, answer a few questions about the user's goal, and get back:

  • Severity-rated findings (critical / high / medium / low, ranked by importance)
  • Heuristic citations — every finding cites a named principle from a controlled vocabulary of 15 frameworks
  • Recommended improvements — one concrete, action-verb-first suggestion per finding
  • Annotated screenshots — numbered, severity-color-coded bounding boxes drawn on the images
  • Structured reports — a self-contained HTML report plus a short Markdown summary, backed by a JSON findings file
  • Scores — six category sub-scores (0–100) plus a calibrated overall score

Unlike Qualia (a SaaS with Supabase edge functions and a Figma plugin), this is a pure local skill: prompts, reference checklists, and a couple of Python scripts for deterministic measurement and image annotation. No backend, no API keys beyond the Claude session itself.

2What the research showed

From the Qualia codebase (production-hardened, worth stealing)

  • Multi-engine analysis beats one big prompt. Qualia runs four scored engines — System Logic & Flow, Heuristic & Navigation, Cognitive & Visual, Interaction Cost — each with its own focus questions and 0–100 calibration anchors (what a 90, a 50, and a 20 look like). Prompts live in supabase/functions/_shared/analyze-prompts.ts and scripts/ux-audit/prompts/*.ts.
  • The ALISSA filter kills false positives. Every candidate finding must pass four gates: Blocker (does it impede the user's goal?), Standard (is this just a normal web pattern? don't flag it), Clutter (does the fix add UI when an existing path exists?), Steelman (why might a competent designer have done this on purpose?). This is almost certainly why the post's author found the output non-generic.
  • Importance = severity × reach. severityWeight(4/3/2/1) × reachBand(1/1.5/2/3) — findings are sorted by impact, not discovery order.
  • Anti-compression calibration. Qualia's prompts explicitly state that clustering scores in 70–89 is a calibration failure and define what 0–49, 50–65, 66–79, 80–89, 90+ mean. Without this, LLMs give everything a polite 78.
  • Accessibility is a separate block, never duplicated inside the engines, and only visually unambiguous violations are flagged (no guessing at focus order from a static image).
  • Bounding boxes on a 0–1000 normalized scale ([ymin, xmin, ymax, xmax]) with validation rules; localized findings must have one, screen-level findings must be null.
  • "One Big Thing" — a single falsifiable root-cause diagnosis, with rules against symptom-chasing and persona-echoing.
  • Strict conciseness contract: issue ≤ 20 words, total finding ≤ 60 words, action verb first, no filler.

From the 15 framework links (defines what's honest to claim)

The frameworks split into three auditability tiers. This is the single most important structural insight: a screenshot-only audit can confirm Tier 1, partially judge Tier 2, and only hypothesize Tier 3. Most "AI design feedback is generic" complaints come from tools that blur these tiers.

TierEvidence availableFrameworks
Tier 1 · StaticSingle screenshot sufficesGestalt, WCAG visual criteria (contrast, use-of-color, target size, labels), Fitts's Law, Hick's Law, Miller's Law, content design heuristics, Norman signifiers/visibility, Cialdini-pattern detection (dark patterns are visible artifacts)

3Framework registry & controlled citation vocabulary

Each framework becomes one reference file with stable, citable IDs. A finding's principle field must match this registry exactly — free-text citations are rejected in the validation pass. This is what makes "the same issue gets flagged the same way every time" (the post's closing point about named categories).

└── fixtures/                       # golden screenshots + expected findings, for calibration

Progressive disclosure keeps context small. SKILL.md stays under ~150 lines and tells Claude which reference files to load for which audit mode and tier — it never inlines the frameworks. A single-screen audit loads only Tier 1 files; a flow audit adds Tier 2; Tier 3 files load only when the user opts into conversion/persuasion analysis.

5Audit workflow

1

Intake — "answer a few questions"

One AskUserQuestion round, max 4 questions: user goal on these screens (free text — drives the Blocker gate), persona (default: first-time visitor), platform (mobile touch / desktop / responsive — changes Fitts targets: 44px vs 24px), and artifact stage (live product / hi-fi mockup / wireframe — calibrates expectations; don't flag placeholder copy in a wireframe). The post's key insight — "the quality of AI feedback depends heavily on how you specify what you are evaluating against" — lives here and in the framework registry.

2

Mode detection

1 image → single-screen mode (Tier 1 + screen-level Tier 2 checks). 2+ images → flow mode (confirm ordering with the user, then add transition checks: consistency across screens, status feedback after actions, peak-end review of the final screen). Each mode has its own prompt section in SKILL.md, mirroring Qualia's SINGLE_SCREEN_PROMPT / FLOW_ANALYSIS_PROMPT split.

3

Deterministic pre-pass (scripts, not vibes)

measure.py runs first and computes hard numbers the model is bad at estimating: WCAG contrast ratios for sampled text/background pairs, approximate target sizes in px, smallest body-text size. Output is injected into the analysis context as contrast_data — exactly Qualia's pattern. The model never invents a contrast ratio; it cites the measured one.

4

Framework passes by tier

Claude reads each screenshot once per category group (classic usability → cognitive laws → trust/persuasion → interaction design → accessibility → content), generating candidate findings with mandatory principle IDs and bounding boxes. Tier 3 checks run last and every Tier 3 finding is forced to type: "hypothesis" with a suggested validation method ("A/B test", "check funnel drop-off at this step").

5

Filter & dedupe

Every candidate passes the four-gate filter (Blocker / Standard / Clutter / Steelman — see filter.md). Cross-framework duplicates merge into one finding with multiple citations (a low-contrast disabled-looking button is one finding citing WCAG 1.4.3 + NRM-02 signifiers, not two). Cap: 5 findings per category, ranked by importance — Qualia's issue cap, which keeps reports readable.

6

Scoring + One Big Thing

Six category sub-scores against written calibration anchors, overall score, and a single falsifiable root-cause diagnosis (rules in §6).

7

Render & deliver

annotate.py draws the boxes → build_report.py produces the self-contained HTML report → serve on 0.0.0.0, verify 200, hand over the Tailscale URL. A ≤60-line Markdown summary is also written for pasting into Linear/GitHub.

6Scoring model

Severity (per finding)

Importance (sort key) = severity weight × reach band, where reach = how many screens/elements the issue touches (≤1 → 1.0, ≤3 → 1.5, ≤9 → 2.0, >9 → 3.0).

Category sub-scores (0–100)

Calibration rule (adopt verbatim from Qualia) Use the full 0–100 range. 0–49 fundamentally broken · 50–65 broken flows, users likely abandon · 66–79 functional but unremarkable · 80–89 strong with minor gaps · 90+ exceptional. Clustering everything in 70–89 is a calibration failure. Each category's reference file includes written anchors for what a 90, a 50, and a 20 look like in that category.

7Finding schema

{
  "id": "F-07",
  "category": "usability",                    // one of the six categories
  "type": "observed" | "hypothesis",          // hypothesis = Tier 3, never moves scores
  "severity": "critical" | "high" | "medium" | "low",
  "reach": 2,                                 // screens/elements affected
  "importance": 4.5,                          // computed: weight × reachBand
  "principles": ["NLS-01", "WCAG 1.4.3"],     // ≥1, validated against registry
  "screen_index": 1,                          // 0-based, null for cross-screen
  "box_2d": [120, 640, 210, 980],             // [ymin,xmin,ymax,xmax] 0–1000; null = screen-level
  "issue": "Primary CTA has 2.8:1 contrast and sits below the fold.",   // ≤20 words
  "why_it_matters": "First-time users scanning for next steps never see it; checkout starts are lost.", // causal chain, ≤2 sentences
  "recommendation": "Raise contrast to ≥4.5:1 and move the CTA above the fold.", // action verb first, ≤25 words
  "evidence": "measured: contrast 2.8:1 (#8a8f98 on #6b7280)",  // from measure.py when available
  "validation": null                          // hypotheses only: how to verify
}

Conciseness contract enforced in a validation pass: total ≤60 words across the three text fields, no filler, no coordinates in user-facing text, bounding box required for localized findings and forbidden for screen-level ones (with the ymin<ymax sanity checks Qualia uses).

One Big Thing

Single structural diagnosis with Qualia's four rules: diagnose the system not the symptom; don't over-attribute (a bad step 9 of 9 usually means the flow is too long); don't echo the persona's language back; be falsifiable ("the flow has too many decision points for a first-time user" is a finding — "there's tension between promise and delivery" is not).

8Annotated screenshots

  • annotate.py (Pillow, stdlib otherwise) takes the original PNGs + findings.json, converts 0–1000 boxes to pixels, draws rounded rectangles color-coded by severity with a numbered badge matching the finding ID, and writes annotated/screen-N.png.
  • Overlap handling: stagger badges, draw highest-severity last so it sits on top.
  • Screen-level findings (null box) are listed in a margin strip, not drawn.
  • The annotated images are base64-embedded in the HTML report so it stays a single portable file.

9Report output

report.html (primary)

Self-contained: scorecard with six category bars + overall, One Big Thing callout, annotated screenshots with click-to-highlight finding cards, findings table sortable by importance, hypotheses section clearly separated, methodology footer listing frameworks applied and which tiers ran. Served at http://clawbot.crane-cod.ts.net:PORT/.

summary.md (secondary)

≤60 lines for Linear/GitHub: overall score, One Big Thing, top 5 findings with citations and recommendations, link to the full report. findings.json ships alongside as the machine-readable source of truth (enables re-audit diffing later).

10Quality controls

  • Four-gate filter on every finding (the single biggest generic-feedback killer, per Qualia).
  • Registry-validated citations — a finding that can't name its principle ID is dropped.
  • Measured, not estimated — contrast and target sizes come from measure.py; the model may not assert numeric WCAG failures without script evidence.
  • Tier honesty — Tier 3 output is labeled hypothesis, excluded from scores, and paired with a validation method.
  • Golden fixtures — 3–5 screenshots with hand-curated expected findings in fixtures/; after any prompt change, re-run and diff (catch both regressions and score drift). Qualia's q-audits meta-skill shows this pays off.
  • No-finding honesty — a clean screen scores high with few findings; the skill must not manufacture issues to look thorough (Standard gate enforces this).

11Implementation phases

1

MVP — single-screen audit, cited & scored

SKILL.md + the 12 framework reference files + scoring.md + filter.md. Intake questions, Tier 1 audit, JSON findings, Markdown report. No scripts yet — accessibility findings limited to non-numeric observations. Validate end-to-end on 2–3 real screenshots. This alone matches most of the LinkedIn post's output.

2

Measurement + annotation

measure.py (contrast, target size) and annotate.py (boxes, badges). Numeric WCAG findings now allowed. Bounding-box validation pass added.

3

Flow mode + HTML report

Multi-screen ordering, Tier 2 transition checks (consistency, status feedback, peak-end), build_report.py + template, Tailscale serving baked into the skill's final step.

4

Calibration & re-audit

Golden fixtures + drift checks; re-audit mode (pass a prior findings.json, get a delta report with fixed/regressed/new); Tier 3 opt-in conversion analysis; the "recurring failures" registry grows from real usage — mirroring how the post's author added custom checks for repeat offenders.
